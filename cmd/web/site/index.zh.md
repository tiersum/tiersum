## 为何选择 TierSum

常见检索会把长文切成大量小块再做相似度匹配，容易**丢失结构与语境**。TierSum 用**分层摘要**（文档 → 章节 → 原文）保留知识结构，并配合**标签与主题**做可解释的导航与筛选，而不是只靠向量「猜」片段边界。

在**热路径**上，**入库时**即可由 AI 写好标签、文档概述与章节级提要，形成后续检索依赖的**「已铺好的」语义层**；**渐进式查询**在此基础上沿 **标签 → 文档 → 章节** 逐步收窄，每一步由大模型打分，像先扫目录再翻到对应小节，路径**可解释、可复查**。**冷路径**则省去大部分入库时 LLM 成本，仍以**整章**建索引与返回；访问频繁时可**晋升**为热文档，获得同样的预摘要能力。

> **冷热一致：以章节为单元。** 无论走「热」还是「冷」，系统都以 **Markdown 章节**（按标题划分的自然段）作为工作与检索粒度，而不是固定长度的随机碎片。摘要、渐进式收窄与冷索引检索都沿这一边界展开，**保证语义从头到尾完整、可理解**。

## 功能特性

### 章节优先的文档处理

传统 RAG 系统将文档切成任意碎片，破坏结构与语境。TierSum 按标题解析 Markdown，形成自然的章节层级，与人类书写和阅读方式一致。

- 标题感知的拆分保留文档结构
- 每章节可配置 token 预算
- 长节支持滑动窗口
- 每个章节拥有稳定的路径标识

### 热 / 冷分层

并非所有文档都需要完整的 LLM 分析。TierSum 允许为每篇文档选择合适的入库路径，在查询质量与成本之间取得平衡。

- **热**：入库时生成完整 LLM 摘要 + 标签
- **冷**：BM25 + 向量混合搜索
- **自动模式**根据内容长度与配额智能选择
- 频繁查询的冷文档自动向热晋升

### 渐进式查询

不同于单一的向量相似度搜索，TierSum 使用多阶段管道模拟人类搜索方式：先按主题收窄，再到文档，最后到章节。

1. **标签筛选** — 从查询中找到相关的目录标签
2. **文档排序** — 用 LLM 相关性对匹配文档打分
3. **章节选择** — 从排序后的文档中挑选最佳章节
4. **可选**：汇总为带引用的回答

### 主题与标签

自动生成的目录标签，按 LLM 策展的主题分组。像人类组织知识一样浏览您的知识库。

### 双轨认证

人类与程序使用独立的认证路径。人类使用浏览器会话（支持通行密钥），程序使用带作用域的 API 密钥。

- 基于角色的访问控制：viewer、user、admin
- API 密钥作用域：read、write、admin
- 持久会话的设备令牌
- 速率限制与暴力破解防护

### REST + MCP

完整的 REST API 用于集成。支持 MCP（模型上下文协议），用于 AI 代理和自动化工作流。

## 工作原理

### 01 — 摄入

上传 Markdown 文档。选择热（完整 LLM 分析）或冷（仅索引）摄入模式。自动模式根据内容长度和配额选择最佳路径。

### 02 — 索引

热文档获得章节级摘要和标签。冷文档拆分为章节，并使用 BM25 + HNSW 向量搜索索引。全部保留标题结构。

### 03 — 查询

用自然语言提问。渐进式查询遍历标签 → 文档 → 章节。获取带引用的综合答案，或浏览原始章节命中结果。

## 快速开始

### 1. 前置条件

- Go 1.23 或更高版本
- Make
- OpenAI API 密钥（或 Anthropic，或本地 Ollama）

### 2. 安装

```bash
# 克隆仓库
git clone https://github.com/tiersum/tiersum.git
cd tiersum

# 复制并编辑配置
cp configs/config.example.yaml configs/config.yaml
# 编辑 configs/config.yaml 并设置您的 LLM API 密钥

# 构建
make build

# 运行
make run
```

### 3. 初始化

在浏览器中打开 http://localhost:8080。完成初始化向导以创建第一位管理员用户。

> 出于安全考虑，初始化端点默认仅可从本地主机访问。

### 4. 第一篇文档

导航到文库页面并点击「添加文档」。粘贴 Markdown 内容并选择摄入模式：

- **自动** — 让 TierSum 根据内容长度和配额决定
- **热** — 强制完整 LLM 分析（查询效果更好，消耗配额）
- **冷** — 仅索引（摄入更快，BM25 + 向量搜索）
