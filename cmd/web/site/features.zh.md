# 功能特性

结构化知识检索所需的一切。TierSum 将分层文档处理、智能分层和混合搜索整合到一个可自托管的平台中。

---

## 章节优先的文档处理

传统 RAG 系统将文档切成任意碎片，破坏结构与语境。TierSum 按标题解析 Markdown，形成自然的章节层级，与人类书写和阅读方式一致。

- 标题感知的拆分保留文档结构
- 每章节可配置 token 预算
- 长节支持滑动窗口
- 每个章节拥有稳定的路径标识

示例文档结构：

```markdown
# 架构决策记录

## 01-why-tier-sum

### 背景
我们需要一个能够保留……的系统

### 决策
使用章节优先的拆分……

### 后果
- 结构得以保留
- 查询返回完整章节
```

## 热 / 冷分层

并非所有文档都需要完整的 LLM 分析。TierSum 允许为每篇文档选择合适的入库路径，在查询质量与成本之间取得平衡。

| 层级 | 描述 | 最适合 |
|------|-------------|----------|
| **热** | 入库时生成完整 LLM 摘要 + 标签 | 频繁查询的文档 |
| **冷** | BM25 + 向量混合搜索 | 大型存档、成本敏感场景 |
| **自动** | 根据内容 + 配额智能选择 | 大多数使用场景 |

- 热：入库时生成完整 LLM 摘要 + 标签
- 冷：BM25 + 向量混合搜索
- 自动模式根据内容 + 配额选择
- 频繁查询的冷文档自动向热晋升

## 渐进式查询

不同于单一的向量相似度搜索，TierSum 使用多阶段管道模拟人类搜索方式：先按主题收窄，再到文档，最后到章节。

**三个阶段：**

1. **标签筛选** — 从查询中找到相关的目录标签
2. **文档排序** — 用 LLM 相关性对匹配文档打分
3. **章节选择** — 从排序后的文档中挑选最佳章节
4. **可选**：汇总为带引用的回答

API 示例：

```http
POST /api/v1/query/progressive
{
  "question": "热/冷分层是如何工作的？",
  "max_results": 10
}

// 响应
{
  "answer": "热/冷分层平衡了……",
  "steps": [
    { "stage": "tags", "matches": 3 },
    { "stage": "documents", "matches": 12 },
    { "stage": "chapters", "matches": 8 }
  ],
  "results": [...]
}
```

## 主题与标签

自动生成的目录标签，按 LLM 策展的主题分组。像人类组织知识一样浏览您的知识库。

- 浏览共享的目录标签
- 标签按*主题*（主题）分组
- 「重归类」根据当前标签目录刷新主题
- 随库增长保持导航意义

## 双轨认证

人类与程序使用独立的认证路径。人类使用浏览器会话（支持通行密钥）。程序使用带作用域的 API 密钥。

**认证方式：**

- **API 密钥** — 程序化访问，带作用域令牌（read / write / admin）
- **浏览器会话** — 用于 Web UI 的 HttpOnly Cookie，基于角色的访问控制
- **通行密钥** — 支持 WebAuthn 的无密码认证

**访问控制：**

- 基于角色的访问：viewer、user、admin
- API 密钥作用域：read、write、admin
- 持久会话的设备令牌
- 速率限制与暴力破解防护

## REST + MCP

完整的 REST API 用于集成。支持 MCP（模型上下文协议），用于 AI 代理和自动化工作流。

- 完整的 REST API，位于 `/api/v1`
- AI 代理的 MCP 协议端点
- OpenAPI 兼容的 JSON 响应
- Prometheus 指标位于 `/metrics`
